# Codex 指令生成规则 v1.2（精简稳定版）

> 适用于采用 `frontend\`、`backend\`、`docs\` 目录结构，并通过架构文档与规则文档约束开发的项目。
> 目标：保证指令结构稳定、输出可审核、执行边界清晰、文档同步可判断。

---

## 1. 固定前缀规则（必须存在）

所有涉及代码修改的 Codex 指令，必须以如下前缀开头：

```text
本指令中：
- 若涉及修改 backend\ 目录下的代码，严格遵循 docs/backend-architecture.md 与 docs/codex-rules.md 的规范。
- 若涉及修改 frontend\ 目录下的代码，严格遵循 docs/frontend-architecture.md 与 docs/codex-rules.md 的规范。
- 若同时涉及修改 backend\ 目录和 frontend\ 目录下的代码，必须分别遵循对应架构规范。
```

此段为强制执行边界声明，不得省略。

---

## 2. 标准结构骨架（栏目固定）

每条 Codex 指令必须包含以下栏目，栏目名称固定：

1. 〖任务名〗
2. 〖任务目标〗
3. 〖实现范围〗
4. 〖接口与口径约束〗（如无则写“无”）
5. 〖强约束（必须遵守）〗
6. 〖非目标（明确不做）〗
7. 〖环境变量 / 配置项〗（如无则写“无”）
8. 〖实现步骤（建议按顺序）〗
9. 〖预期文件清单（新增/修改）〗
10. 〖文档同步要求〗（如无则写“无”）
11. 〖验收步骤（由 Codex 执行）〗
12. 〖输出要求（本次任务专用，用于审核）〗

---

## 3. 核心执行原则

### 3.1 骨架固定，内容弹性

* 栏目结构必须稳定。
* 栏目内部内容可根据任务类型扩展。
* 不允许新增随意的一级栏目。
* 不得删减固定栏目。

---

### 3.2 文档同步要求必须显式判断

所有涉及代码、配置、接口、数据模型、路由、组件、测试或架构决策变更的 Codex 指令，都必须判断是否需要同步文档。

文档同步要求的执行规则如下：

* 若需要同步，必须明确列出需要更新的文档。
* 若不需要同步，必须在“〖文档同步要求〗”栏目中写“无”，不得省略。
* 同步范围可包括但不限于 `docs\` 下的架构文档、规则文档、测试文档，以及 `docs\handoff\` 下已经存在的交接文档。
* 不得要求创建与本次任务无关的文档。
* 不得假设某个 `docs\handoff\` 文件一定存在；生成具体指令时，必须以当前仓库实际存在的文件为准。

---

### 3.3 输出要求必须与任务绑定（关键原则）

“输出要求”不得使用固定模板，必须根据本次任务的风险点、审核重点和边界信息定制。

输出要求至少包含：

* 新增/修改文件清单
* 本任务的关键实现说明或关键代码片段（视任务类型而定）
* 文档同步情况
* 根据任务实际情况明确声明未涉及 `backend\`、未涉及 `frontend\`、未新增依赖、未修改接口、未修改配置等边界信息
* 逐项区分 Codex 已实际执行并通过、Codex 已实际执行但失败、因权限 / 工具 / 环境限制未执行，以及需要开发者完成的特定人工项

上述边界声明必须按任务实际情况定制，不要求机械输出全部声明。

仅提供命令文本但未实际执行，必须报告为未执行，不得记为通过；人工待签收、工具限制或其他 `not_executed` 状态也不得记为通过。

输出要求是本规则中最重要的审核锚点。

---

### 3.4 数据库任务声明（强制）

凡任务涉及数据库选择或隔离治理，或者会连接数据库、启动会连接数据库的应用或测试进程、准备或清理数据库 fixture，Codex 指令都必须完整声明以下字段；用途为 `none` 时也不得省略：

```text
数据库用途类别：
具体数据库映射来源：
允许数据库：
禁止数据库：
允许操作：
是否允许写入：
是否允许 cleanup：
是否需要长期保留 namespace：
进程与环境隔离要求：
连接后的实际数据库名门禁：
```

声明规则：

1. `数据库用途类别` 必须使用 `docs/codex-rules.md` 定义的 `none`、`development`、`standard_test`、`browser_acceptance` 或 `production_or_operations`。
2. `具体数据库映射来源` 必须指向项目 testing playbook、用户明确输入或其他明确项目文档，不得依据经验猜测。
3. `允许数据库` 和 `禁止数据库` 必须填写具体项目值，不能只写“测试库”“开发库”或其他模糊称呼。
4. `none` 类任务必须明确填写“不连接数据库”，并禁止无故启动应用或数据库相关进程。
5. 混合任务必须按独立子进程分别声明，例如普通 unit / E2E、Browser fixture、Browser test backend；每个子进程分别填写用途、允许数据库、禁止数据库、操作和门禁。同一任务同时涉及 `standard_test` 与 `browser_acceptance` 时，必须明确声明至少两个独立子进程：`standard_test` 子进程只能加载 `backend/.env.test`，`browser_acceptance` 子进程只能加载 `backend/.env.browser-acceptance`。两个文件可以同时存在于磁盘，但不得叠加加载到同一进程，也不得依赖 dotenv 加载顺序、shell 旧变量或后加载覆盖行为决定连接数据库。
6. 生成本地隔离测试任务指令时，可以直接引用项目 testing playbook 约定且被 Git 忽略的本地凭据文件，或声明由同一隔离父进程稳定注入目标环境变量，并要求 Codex 自动把固定测试凭据传递给对应的独立目标子进程或测试数据构造流程；指令只能声明凭据来源、变量职责、注入方式和目标进程，不得包含或复制实际密码。
7. 只要任务包含 Browser fixture 的 prepare、replace、prepared verify、Browser 登录或角色 Session、post-browser verify，指令就必须明确声明测试应用账号密码的稳定来源、同一 namespace 生命周期内保持同一个密码值，以及所有实际创建、认证或校验账号密码的独立子进程如何接收该凭据；不校验账号密码的 cleanup 无需人为增加该依赖。指令必须禁止每条命令临时生成或派生密码、使用时间或进程相关值、从数据库 URI、数据库用户密码或无关 Secret 派生、因父进程缺少变量而临时构造新值，以及在不同阶段注入不同值；凭据不得通过 CLI 参数传递。若已准备的 namespace 与当前稳定凭据不一致，指令必须要求使用正确固定凭据显式执行受控 replace，随后在完全相同的稳定凭据环境中执行只读 prepared verify，并仅在通过后进入 Browser；不得把该错误写成需要逐次人工输入、逐次审批或重新生成凭据，也不得以反复 verify、手工改库、降低校验标准或虚报 prepared gate 通过来绕过。
8. 不得复用已设置数据库连接的 shell 上下文，也不得通过一个全局环境配置同时服务不同数据库用途。切换用途或复用 shell 前，应优先启动独立进程，或明确清除或覆盖数据库主连接变量、管理连接变量、项目定义的数据库用途变量及其他用途相关变量。具体变量名必须从项目级 testing playbook、项目环境约定、任务声明或用户明确输入中解析；通用指令规范不得硬编码项目专属变量名。这是数据库隔离门禁，不是要求每一步清空固定测试密码的保密仪式。
9. `standard_test` 与 `browser_acceptance` 中只能访问本地隔离目标测试数据且不被其他环境复用的固定凭据，包括数据库用户密码、fixture / E2E / Browser 验收创建的测试应用用户密码和其他确定性测试账号密码，按普通测试凭据处理并默认自动使用。除非用户明确另有要求，指令不得机械要求剪贴板授权、一次性密码、每次重新生成、Secret Manager、人工输入、人工交互或每次向用户询问。
10. 数据库密码、完整连接串和其他 Secret 默认不得包含在指令中。符合上一条的固定测试凭据可以来自项目约定的本地忽略文件，或由同一隔离父进程按声明的职责稳定注入目标子进程，但不得复制到 Git 跟踪文件、文档、日志、manifest、截图、生成物、最终报告或提交记录。该有限规则不得用于 `development`、`production_or_operations`、真实业务数据、其他非目标数据库或可访问非隔离环境 / 真实数据的凭据；这些凭据和未经授权的其他 Secret 仍按严格保护对象处理。
11. 输出要求必须报告各子进程的实际数据库用途和实际数据库名校验结果，但不得输出密码、完整连接串或其他 Secret；即使自动使用本地隔离测试专用固定凭据，最终报告也不得回显其值。
12. 如无法唯一解析用途或具体数据库，或连接后的实际数据库名与声明不一致，指令必须要求立即停止，不得自动回退到其他数据库。

---

### 3.5 验收执行主体与自动化责任（强制）

1. Codex 是默认验收执行主体。当前工具、权限和任务边界允许完成的全部验收步骤，必须由 Codex 实际执行，包括指令要求的 lint、typecheck、build、unit、E2E、fixture 生命周期操作、应用启动与健康检查、可可靠完成的 Browser 自动化，以及 Git diff、status 和范围检查。
2. 指令不得只列出命令、步骤或建议并默认要求开发者手动执行，也不得把整个验收阶段作为人工操作清单交还开发者。
3. 开发者批准权限提示、确认高风险命令或提供外部授权，只是 Codex 执行该步骤的前置条件，不改变执行主体；获得批准后，指令必须要求 Codex 继续执行。
4. 权限审查超时、命令未启动或执行许可未获得时，该步骤必须记为 `not_executed`，不得记为失败或通过；Codex 应请求必要授权，并在获得授权后重新执行，不得要求开发者复制命令到终端。
5. 命令已启动但退出码非 0 时属于实际执行失败。仅“已请求执行”“已获授权”或“已提供命令”均不构成验收通过。
6. 只有当前工具无法支持的实体设备操作、无法可靠产生或观察的原生交互、必须人工完成的主观视觉 / 听觉 / 临床判断、Codex 无法代替的外部登录 / 验证码 / 硬件 / 安全审批动作，或取得授权后仍因环境缺少必要工具或权限而无法自动执行的具体单项，才允许标记为需要开发者操作或授权。
7. 每个人工例外必须逐项说明 Codex 无法执行的原因、开发者需要完成的具体动作、需要返回的脱敏结果及该项的准确状态；不得因此把其他可自动化步骤一并转交开发者。
8. 人工结果返回后，Codex 仍负责继续完成后续可自动化验证和整体收口；不得用静态检查、模拟结果或文字声明替代确实需要人工完成的验收。

---

## 4. 不允许的行为

* 不得绕过既有架构规范（backend / frontend 架构文档）
* 不得擅自新增第三方依赖
* 不得跨目录越界修改（除非任务明确允许）
* 不得要求创建与本次任务无关的文档
* 不得要求输出每个文件的完整最终内容

---

## 5. 使用方式

在任意新会话中，如需生成 Codex 指令，可直接说明：

> “按仓库中的 `docs/codex-instruction-spec.md` 生成 Codex 指令。”

模型必须按本规则输出。

---

## 6. 版本说明

当前版本：v1.2（精简稳定版）
适用场景：适用于需要通过稳定指令结构控制修改范围、验证过程与文档同步的持续开发项目
v1.2 核心变化：固定验收栏目改为由 Codex 执行，并明确自动化执行责任及人工例外边界
升级条件：当仓库结构、治理方式或审核要求发生明显变化时再讨论规则升级
