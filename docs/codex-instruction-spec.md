# Codex 指令生成规则 v1.1（精简稳定版）

> 适用于采用 `frontend\`、`backend\`、`docs\` 目录结构，并通过架构文档与规则文档约束开发的项目。
> 目标：保证指令结构稳定、输出可审核、执行边界清晰、文档同步可判断。

---

## 1. 固定前缀规则（必须存在）

所有涉及代码修改的 Codex 指令，必须以如下前缀开头：

```text
本指令中：
- 若涉及修改 backend\ 目录下的代码，严格遵循 docs/backend-architecture.md 与 docs/codex-rules.md 的规范。
- 若涉及修改 frontend\ 目录下的代码，严格遵循 docs/frontend-architecture.md 与 docs/codex-rules.md 的规范。
- 若同时涉及修改 backend\ 目录和 frontend\ 目录下的代码，必须分别遵循对应架构规范。
```

此段为强制执行边界声明，不得省略。

---

## 2. 标准结构骨架（栏目固定）

每条 Codex 指令必须包含以下栏目，栏目名称固定：

1. 〖任务名〗
2. 〖任务目标〗
3. 〖实现范围〗
4. 〖接口与口径约束〗（如无则写“无”）
5. 〖强约束（必须遵守）〗
6. 〖非目标（明确不做）〗
7. 〖环境变量 / 配置项〗（如无则写“无”）
8. 〖实现步骤（建议按顺序）〗
9. 〖预期文件清单（新增/修改）〗
10. 〖文档同步要求〗（如无则写“无”）
11. 〖验收步骤（由开发者本地执行）〗
12. 〖输出要求（本次任务专用，用于审核）〗

---

## 3. 核心执行原则

### 3.1 骨架固定，内容弹性

* 栏目结构必须稳定。
* 栏目内部内容可根据任务类型扩展。
* 不允许新增随意的一级栏目。
* 不得删减固定栏目。

---

### 3.2 文档同步要求必须显式判断

所有涉及代码、配置、接口、数据模型、路由、组件、测试或架构决策变更的 Codex 指令，都必须判断是否需要同步文档。

文档同步要求的执行规则如下：

* 若需要同步，必须明确列出需要更新的文档。
* 若不需要同步，必须在“〖文档同步要求〗”栏目中写“无”，不得省略。
* 同步范围可包括但不限于 `docs\` 下的架构文档、规则文档、测试文档，以及 `docs\handoff\` 下已经存在的交接文档。
* 不得要求创建与本次任务无关的文档。
* 不得假设某个 `docs\handoff\` 文件一定存在；生成具体指令时，必须以当前仓库实际存在的文件为准。

---

### 3.3 输出要求必须与任务绑定（关键原则）

“输出要求”不得使用固定模板，必须根据本次任务的风险点、审核重点和边界信息定制。

输出要求至少包含：

* 新增/修改文件清单
* 本任务的关键实现说明或关键代码片段（视任务类型而定）
* 文档同步情况
* 根据任务实际情况明确声明未涉及 `backend\`、未涉及 `frontend\`、未新增依赖、未修改接口、未修改配置等边界信息

上述边界声明必须按任务实际情况定制，不要求机械输出全部声明。

输出要求是本规则中最重要的审核锚点。

---

### 3.4 数据库任务声明（强制）

凡任务涉及数据库选择或隔离治理，或者会连接数据库、启动会连接数据库的应用或测试进程、准备或清理数据库 fixture，Codex 指令都必须完整声明以下字段；用途为 `none` 时也不得省略：

```text
数据库用途类别：
具体数据库映射来源：
允许数据库：
禁止数据库：
允许操作：
是否允许写入：
是否允许 cleanup：
是否需要长期保留 namespace：
进程与环境隔离要求：
连接后的实际数据库名门禁：
```

声明规则：

1. `数据库用途类别` 必须使用 `docs/codex-rules.md` 定义的 `none`、`development`、`standard_test`、`browser_acceptance` 或 `production_or_operations`。
2. `具体数据库映射来源` 必须指向项目 testing playbook、用户明确输入或其他明确项目文档，不得依据经验猜测。
3. `允许数据库` 和 `禁止数据库` 必须填写具体项目值，不能只写“测试库”“开发库”或其他模糊称呼。
4. `none` 类任务必须明确填写“不连接数据库”，并禁止无故启动应用或数据库相关进程。
5. 混合任务必须按独立子进程分别声明，例如普通 unit / E2E、Browser fixture、Browser test backend；每个子进程分别填写用途、允许数据库、禁止数据库、操作和门禁。
6. 不得复用已设置数据库连接的 shell 上下文，也不得通过一个全局环境配置同时服务不同数据库用途；连接信息必须按子进程从受控环境变量注入。
7. 指令不得包含数据库密码、完整连接串或其他 Secret。
8. 输出要求必须报告各子进程的实际数据库用途和实际数据库名校验结果，但不得输出 Secret。
9. 如无法唯一解析用途或具体数据库，或连接后的实际数据库名与声明不一致，指令必须要求立即停止，不得自动回退到其他数据库。

---

## 4. 不允许的行为

* 不得绕过既有架构规范（backend / frontend 架构文档）
* 不得擅自新增第三方依赖
* 不得跨目录越界修改（除非任务明确允许）
* 不得要求创建与本次任务无关的文档
* 不得要求输出每个文件的完整最终内容

---

## 5. 使用方式

在任意新会话中，如需生成 Codex 指令，可直接说明：

> “按仓库中的 `docs/codex-instruction-spec.md` 生成 Codex 指令。”

模型必须按本规则输出。

---

## 6. 版本说明

当前版本：v1.1（精简稳定版）
适用场景：适用于需要通过稳定指令结构控制修改范围、验证过程与文档同步的持续开发项目
升级条件：当仓库结构、治理方式或审核要求发生明显变化时再讨论规则升级
